# Service 11.12 update

## Requests - Add Relation rules

You can create hierarchical and side‑by‑side relationships (Related, Parent or Child), helping you understand dependencies, track progress and manage complex service scenarios with confidence. This gives you better control, improved visibility and a smoother way to work with interconnected requests.

Read more about [how to use Related requests][2] in SuperOffice Service.

There are three default relations that cannot be edited or deleted, but you can add additional request relations to support your workflow in SuperOffice Service.

![Admin Requests - Add a new request relation and add the correct settings  -screenshot][img1]

When creating a new relation, you can assign it to a specific request type and define what type of relation it is. You also need to select the related request types it applies to. In addition, you can configure several on/off options on what should happen on closing related request. Add a clear description so your users easily understand how and when to use this relation.

Learn more about [how to set up your SuperOffice Service][1].

## Marketing - Form field restrictions

Our customers tend to use SuperOffice Forms to register newsletter recipients, and/or sign up/reach out to be contacted. Standard fields like company "Business" and person "Interest" fields are important part of their segmentation work and reaching out to these contacts afterwards.

Possibility for extra tables and UDEF support in Forms: The possibility of using lists defined in admin within a form element dropdown is fine, however the real use case from customers is to be able to update UDEF on persons from the form submission itself.

Additional standard fields requested for Person card:

* private number
* preferred language

* birthdate
* address
* city
* post code
* language

Additional standard fields requested for Company:

* department
* business
* number
* main contact

Background: As we do not know what kind of data our customers are keeping in their custom fields, we need to make sure their data quality and integrity is kept. Today it is up to every form user to consider if a field should only be updated when empty, or if overwriting existing information should be possible. When offering more fields, and potentially more essential fields keeping important information, the risk of leaving this to the form designer will increase. We need to include a way of mitigating this risk, for example making a crm administrator consider the risks involved in allowing access and overwriting possibilities to unauthorized contacts through a form.

The sketch below is communicating the risks to the crm administrator, and gives the administrator a control panel for removing field access.

All fields made available within the Form builder have by default an update policy where only empty fields are updated.  Users building the form have the possibility to allow overwrite. This is as today. Users might for instance select to allow overwrite on contact title to keep the field updated. They do this by checking the checkbox “replace” on the field configuration within the form.

When custom fields are available in the form builder, administrators should be able to add restrictions for fields where overwrite should never be possible. The user building the form will then not be able to allow overwrite for such a field (Checkbox “replace” not available).

Administrator should also be able to remove fields from the form builder when they should not be included in any form at all.
