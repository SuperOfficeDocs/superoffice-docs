# Settings and maintenance 11.12 update

## Requests - Add Relation rules

You can define relation rules to control how linked requests behave when using request relations. These rules are  for parent–child relationships in more complex service workflows.

Relation rules let you standardize how requests are connected and ensure consistent handling when requests are created or closed.
What are relation rules?

Relation rules apply when specific request types are linked together (for example, when one request is set as the parent of another). They allow you to define automatic behavior in the request UI, helping agents follow the intended workflow.

![Admin Requests - Add a new request rule and add the correct condition and behavior  -screenshot][img1]

There are three default relations types (Parent, Child, Related) that cannot be edited or deleted, but you can add additional rules to support your workflow in SuperOffice Service.

When creating a rule, you define:

* Source request type (e.g. Child)
* Relation type (Parent–Child)
* Target request type (e.g. Parent)

This lets you control behavior for specific combinations of request types.

When creating a new relation rule, you can add conditions From (source) and To (target) of Request types with a specific Relation type, where the relation is either Parent or Child. You also add different behavior of the rule like "require parent to be set" and "Prevent parent closure". The rule triggers automatic actions in the request user interface only.

This is to support complex scenarios like you have an incident in your system, and you get multiple requests coming in due to this incident. If you set a rule on the request type "Incident" as Parent for "Support requests", you have the ability to set on the "Offer to close children and send messages when parent is closed".

This means that when the incident is fixed, you can send a message to all child requests with information and also close all the requests.

In **SuperOffice Service** you can create hierarchical and side‑by‑side relationships (Related, Parent or Child) of requests, helping you understand dependencies, track progress and manage complex service scenarios with confidence. This gives you better control, improved visibility and a smoother way to work with interconnected requests.

## Alt 2: Requests – Add relation rules

You can define relation rules to control how linked requests behave when using request relations. These rules are used for parent–child relationships in more complex service workflows and help you standardise how connected requests are handled.

### What are relation rules?

Relation rules apply when specific request types are linked together (for example, when one request is set as the parent of another). They define automatic behaviour in the request UI, ensuring that agents follow the intended workflow.

SuperOffice Service includes three default relation types—Parent, Child and Related—which cannot be edited or deleted. However, you can add additional rules to support your business processes.

![Admin Requests - Add a new request rule and add the correct condition and behavior  -screenshot][img1]

### Creating a relation rule

When creating a rule, you specify:

* Source request type (for example, Child)
* Relation type (Parent–Child)
* Target request type (for example, Parent)

You can then define:

* Conditions for From (source) and To (target) request types
* Behaviour such as:
  * Require parent to be set
  * Prevent parent closure
  * Offer to close children and send messages when parent is closed

These rules trigger automatic actions only in the request user interface.

### Example scenario

If you have an "Incident" request type that acts as a Parent for multiple Support request types, you can apply a rule such as Offer to close children and send messages when parent is closed.
When the incident is resolved, the system can automatically close all related child requests and notify the affected customers.

### Why use relation rules?

Relation rules help you:

* Manage hierarchical or parallel relationships between requests
* Understand dependencies
* Track progress more effectively
* Work efficiently with interconnected or complex service scenarios

This gives you better control, improved visibility and a smoother workflow in SuperOffice Service.

## Alt 3: Requests - Set up Relation rules

You can define relation rules to control how linked requests behave when using request relations. These rules are  for parent–child relationships in more complex service workflows.

Relation rules let you standardize how requests are connected and ensure consistent handling when requests are created or closed.

### What are relation rules?

Relation rules apply when specific request types are linked together (for example, when one request is set as the parent of another). They allow you to define automatic behavior in the request UI, helping agents follow the intended workflow.

![Admin Requests - Add a new request rule and add the correct condition and behavior  -screenshot][img1]

### How relation rules work

When creating a rule, you define:

* Source request type (e.g. Child)
* Relation type (Parent–Child)
* Target request type (e.g. Parent)

This lets you control behavior for specific combinations of request types.

Available rule options
You can enable one or more of the following behaviors:

* Require parent on creation
* Forces users to set a parent request when creating a request of a specific type.

**Example:**
Ensure all Support requests are linked to a Master or Generic request.

Prevent parent closure
Prevents closing a parent request until all child requests are closed.

### Use case

Ensure all related work is completed before resolving the main request.

#### Offer to close parent

Prompts the user to close the parent request when all child requests are closed.

#### Offer to close children with message

When closing a parent request, users are prompted to:

* Close all child requests.
* Send a predefined message to each.

### Typical use case

When a root issue (like a bug or incident) is resolved, all related requests can be closed with consistent communication.

### When should you use relation rules?

Relation rules are advanced configuration and typically apply to more structured support setups.
Most teams will just use request relations (Parent, Child, Related) without rules

* Rules are mainly useful for standardization and automation in larger or more complex workflows
* They are especially relevant in scenarios like:
* Incident management (master ticket with linked requests)
* Bug tracking across multiple customers
* Structured support processes with strict dependencies
